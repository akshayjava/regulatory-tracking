## 2024-05-18 - Missing ARIA Labels in Unlabeled Form Inputs and Improved Empty States
**Learning:** This app uses inline styles heavily and inputs like Search and Status filters did not have associated `<label>` tags. This caused accessibility issues for screen readers. Additionally, replacing generic "No results found" messages with a clear recovery path (e.g., a CTA to clear filters) significantly improves the user experience.
**Action:** Next time working on custom styled inputs, ensure ARIA attributes are added if standard label tags aren't present. Also, look out for empty states that leave the user stranded, and try to add actionable buttons to help them recover.## 2024-05-17 - Focus states for inline-styled React components
**Learning:** The frontend React application heavily relies on inline styles instead of external CSS classes or utility frameworks. Because CSS pseudo-classes like `:focus-visible` cannot be natively applied via inline styles, standard keyboard accessibility focus rings are often completely missing on interactive elements.
**Action:** When implementing or refactoring interactive elements (like buttons) in this app, manually handle accessibility focus states by attaching `onFocus` and `onBlur` handlers to semantic elements to render custom focus rings (e.g., via `e.currentTarget.style.boxShadow = '0 0 0 2px #3b82f6'`).
