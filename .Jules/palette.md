## 2024-05-18 - Missing ARIA Labels in Unlabeled Form Inputs and Improved Empty States
**Learning:** This app uses inline styles heavily and inputs like Search and Status filters did not have associated `<label>` tags. This caused accessibility issues for screen readers. Additionally, replacing generic "No results found" messages with a clear recovery path (e.g., a CTA to clear filters) significantly improves the user experience.
**Action:** Next time working on custom styled inputs, ensure ARIA attributes are added if standard label tags aren't present. Also, look out for empty states that leave the user stranded, and try to add actionable buttons to help them recover.
## 2024-05-19 - Semantic Buttons for Clickable Divs and Inline Focus Styles
**Learning:** This app heavily relies on inline styles (`style={{...}}`) instead of external CSS. Consequently, pseudo-classes like `:focus-visible` cannot be used directly in styles. When replacing non-semantic interactable `div` elements with proper `<button>` elements, standard browser focus rings might break the design or be invisible due to complex backgrounds.
**Action:** When fixing non-semantic `div` buttons in this app, convert them to `<button>` elements (with `role` and `aria-*` attributes as needed) and manually implement focus indicators by attaching `onFocus` and `onBlur` handlers to manage component state, conditionally applying inline styles (like `boxShadow`) for a visible focus ring.
