## 2024-05-18 - Missing ARIA Labels in Unlabeled Form Inputs and Improved Empty States
**Learning:** This app uses inline styles heavily and inputs like Search and Status filters did not have associated `<label>` tags. This caused accessibility issues for screen readers. Additionally, replacing generic "No results found" messages with a clear recovery path (e.g., a CTA to clear filters) significantly improves the user experience.
**Action:** Next time working on custom styled inputs, ensure ARIA attributes are added if standard label tags aren't present. **Learning:** The Analysis component used clickable `<div>` elements for its checklist items, causing accessibility issues because screen readers and keyboard users could not interact with them as proper interactive elements. Replacing them with `<button role="checkbox">` significantly improves accessibility. Because inline styles are heavily used, CSS pseudo-classes like `:focus-visible` cannot be natively applied, so adding `onFocus` and `onBlur` to manage a `boxShadow` focus ring restores critical visual keyboard feedback.
**Action:** Always prefer semantic HTML elements like `<button>` over clickable `<div>` elements for interactive components. When working in an architecture constrained to inline styles, ensure visual focus states are manually handled via JavaScript event listeners.
