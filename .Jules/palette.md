## 2024-05-18 - Missing ARIA Labels in Unlabeled Form Inputs and Improved Empty States
**Learning:** This app uses inline styles heavily and inputs like Search and Status filters did not have associated `<label>` tags. This caused accessibility issues for screen readers. Additionally, replacing generic "No results found" messages with a clear recovery path (e.g., a CTA to clear filters) significantly improves the user experience.
**Action:** Next time working on custom styled inputs, ensure ARIA attributes are added if standard label tags aren't present. Also, look out for empty states that leave the user stranded, and try to add actionable buttons to help them recover.
## 2026-05-16 - Accessible Checkboxes with Inline Styles
**Learning:** When dealing with custom checklist items that use inline styles (meaning pseudo-classes like `:focus-visible` are unavailable), standard `<div>` elements with `onClick` handlers are completely inaccessible to keyboard and screen reader users. Simply changing them to `<button>` elements with `role="checkbox"` and `aria-checked` states improves screen reader semantics, but does not solve visual keyboard accessibility without CSS.
**Action:** Always map interactive checklist/toggle `<div>` elements to semantic `<button>` tags with the `checkbox` role. Furthermore, explicitly handle visual focus states for inline-styled apps by attaching `onFocus` and `onBlur` handlers directly to the `<button>` to render an accessible focus ring via `boxShadow` styling.
